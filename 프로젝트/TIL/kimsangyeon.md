## 2026.01.16

### 오늘 한 일

* **스마트 카트 키오스크 웹 프로젝트 초기 세팅:** Flask + Vue.js + Vuetify 환경 구축
* **UI/UX 디자인:** Figma를 활용한 키오스크 화면 디자인 초안 진행
* **데이터베이스 구축:** 서비스 시나리오에 맞는 DB 구조 생성
* **기술 의사결정 및 검토:** 자율주행 데이터 아키텍처 설계, 모터 / RFID 등 하드웨어 관련 논의 진행, 웹 프론트엔드 구조 확정

### 정리

#### 기술적 의사결정 및 배운 점

1. **자율주행 데이터 아키텍처 (Hybrid Approach)**
* **좌표 데이터 이원화:** 맵 틀어짐에 따른 유지보수 비용을 줄이기 위해 데이터를 분리함.
* **DB:** 논리적 위치 ID (예: `PRODUCT_A2` - 사과 매대) 추가 관리. 오린카 여러대가 있는 상황 가정하여 DB에서 위치를 관리한다.

2. **RFID/NFC 하드웨어 선정**
* **NFC 활용 가능성:** NFC가 13.56MHz RFID 기술의 부분집합임을 확인. 별도 리더기 없이 팀원 스마트폰(NFC 모드)과 앱을 이용해 태그 쓰기(Write)가 가능하여 개발 비용 절감.

3. **웹 개발 구조 (Flask + Vue.js)**
* **MPA + SPA 장점 결합:** Flask 템플릿 상속(`base.html`)을 기본으로, `Vue.js`를 얹어 부드러운 화면 전환과 제스처(Swipe) 기능을 구현.
* **Vue.js 선택 이유:** 단순 HTML/JS 대비 데이터 바인딩(Reactivity)이 강력하여 장바구니 수량/총액 실시간 업데이트 구현에 유리함.

* **작업 지시서 작성 (.yml):** 단순히 코드만 올리는 게 아니라, "어떤 순서로 배포할지"를 `.gitlab-ci.yml`에 코드로 적어두면 알아서 실행된다는 점이 인상적이었음.

* **Polling (주기적 확인):** 서버에 설치된 `gitlab-runner`라는 프로그램이 켜져 있으면서, 주기적으로 GitLab에게 "나 시킬 일 있어?" 하고 계속 물어본다.
* **Job 할당:** 내가 `Push`를 하면 GitLab이 "배포 작업(Job)이 생겼어!"라고 Runner에게 알려준다.
* **Fetch & Checkout:** Runner는 미리 받아둔 **`CI_JOB_TOKEN` (임시 출입증)**을 사용해 권한 인증을 통과하고, 자동으로 `git fetch`와 `checkout`을 수행해 최신 코드를 서버로 당겨온다. (이 과정 덕분에 비밀번호를 칠 필요가 없음)

#### 겪은 문제 및 해결 방법 (Troubleshooting)

**Flasgger(Swagger) YAML 파싱 에러 (`mapping values are not allowed`)**

* **상황:** 서버 배포 후 Swagger 문서 생성 과정에서 `yaml.scanner.ScannerError`가 발생하며 서버가 실행되지 않음.
* **원인:** YAML 파서가 trailing whitespace를 문자열의 일부로 인식하여 예상치 못한 결과를 낼 수 있음.
1. 주석(`docstring`) 내 `description` 부분에서 콜론(`:`) 뒤에 공백과 문자가 오자, 이를 단순 문자열이 아닌 **새로운 Key-Value 쌍**으로 잘못 인식함.
2. 일부 라인 끝에 **불필요한 공백(Trailing Whitespace)**이 남아있어 들여쓰기(Indentation)에 민감한 YAML 파서가 오작동함.

* **해결:**
1. YAML 파서가 혼동하지 않도록 해당 문자열을 **쌍따옴표("")**로 감싸 명시적인 문자열임을 선언함.
* 예시: `description: "위치 코드 (예: PARK-001)"`

2. IDE 설정을 통해 **줄 끝 공백(Trim Trailing Whitespace)**을 일괄 제거

#### 배운 점 (Tech Note)

**리버스 프록시(Reverse Proxy) 아키텍처와 포트 바인딩**

* **작동 원리:**
1. **외부(Client):** 포트 생략 시 **기본 포트**로 요청 → AWS 방화벽 통과(Allow).
2. **중간(Nginx):** 80번으로 들어온 요청을 받아, 내부의 **5001번(Flask)**으로 토스(Forwarding).
3. **내부(Flask):** 외부와 직접 통신하지 않고 Nginx가 건네준 요청을 5001번에서 처리.
* **결론:** 프론트 url 주소에 `:5001`을 명시하는 것은 방화벽에 막힌 '내부 포트'으로 직접 진입하려는 시도여서 실패하며, 포트를 생략해야 '정문(80번, Nginx)'을 통해 안전하게 연결됨.

**Intersection Observer API의 `rootMargin`**`

* **개념:** 요소가 뷰포트(화면)에 진입하는 것을 감지할 때, **감지 영역을 확장하거나 축소**하는 속성. (CSS의 `margin`과 유사)
* **활용:** 무한 스크롤(Infinite Scroll) 구현 시, 사용자가 스크롤을 끝까지 내리기 전(예: 바닥에서 200px 위)에 미리 다음 데이터를 불러오고 싶을 때 사용.
* **설정:** `rootMargin`을 설정하면, 실제 뷰포트보다 아래쪽으로 더 넓은 영역을 감지 범위로 인식하여 **미리 로딩(Pre-fetching)** 하는 부드러운 UX를 구현할 수 있음.

* **Waypoint Navigation:** 단순 단일 좌표 이동이 아닌, `followWaypoints` API를 활용하여 여러 경유지를 순차적으로 방문하는 순찰(Patrol) 로직 설계.
* **Costmap Layer:** 정적 지도(Static) 위에 실시간 센서(Obstacle)와 안전지대(Inflation)가 겹쳐지는 원리를 파악하고, 이를 통한 돌발 장애물 회피 원리 학습.

#### 겪은 문제 및 해결 방법 (Troubleshooting)

**Gazebo 실행 시 `process has died` (Ghost Robot)**
* **원인:** 종료 후에도 백그라운드에 `gzserver` 프로세스가 살아있어 포트 충돌 발생.
* **해결:** `killall -9 gzserver gzclient` 명령어로 좀비 프로세스를 정리하고, 이를 `.bashrc`에 `kg` 단축키로 등록하여 간편하게 사용 가능.

#### 배운 점 (Tech Note)

**싱글톤 패턴 (Singleton Pattern)**
클래스의 인스턴스를 모듈 레벨에서 하나만 생성하여 export 함으로써, 중복 연결을 방지하고 전역에서 일관된 소켓 상태를 관리하는 효율적인 패턴임을 확인함.

**실전과 시뮬레이션의 차이 (Loop Structure)**
시뮬레이션 코드는 1회성 실행(`exit`)이지만, 실제 서비스 코드는 Nav2 서버를 끄지 않고 루프 상태에서 외부 명령을 대기하는 구조로 설계해야 함을 인지함.

### WSL2 기반 Gazebo-RViz 동시 실행 시 GPU 성능 저하 이슈

#### 1. 문제 현상 (Resource Contention)

* **GPU 가속 불균형:** 단독 실행 시에는 원활하나, Gazebo와 RViz를 함께 띄우면 프레임 속도(FPS)가 급격히 하락하고 시스템이 버벅임.
* **DirectX 12 매핑 병목:** WSL2는 GPU 하드웨어를 직접 쓰는 게 아니라 **D3D12 가속 레이어**를 거쳐 GPU를 공유함. 이때 두 GUI 앱이 동시에 GPU를 점유하려고 시도하면서 하드웨어 가속 대신 CPU 렌더링 비중이 늘어나고 병목이 발생함.

#### 2. 주요 원인

* **WSLg 렌더링 오버헤드:** Wayland 및 X11 앱을 윈도우 창으로 변환하는 과정에서 GPU 연산 부하가 가중됨.
* **VRAM 및 대역폭 제한:** 가상화 환경 특성상 실제 하드웨어의 모든 대역폭을 100% 활용하지 못해 다중 3D 렌더링 앱 구동 시 병목 현상이 고착화됨.

#### 3. 대응 방안 및 시도 내역

* **Mesa 드라이버 강제 최적화:** `MESA_D3D12_DEFAULT_ADAPTER_NAME=NVIDIA` 설정을 통해 통합 그래픽이 아닌 외장 GPU를 명시적으로 사용하도록 시도. 해결은 못함.
* **소프트웨어 렌더링 전환 (최후의 수단):** GPU 경합이 너무 심해 크래시가 날 경우, `LIBGL_ALWAYS_SOFTWARE=1`을 설정해 차라리 CPU 연산으로 분산을 시도했으나 해결 못함.

### 느낀 점

모든 트러블이 단순 검색과 AI로 찾은 걸로 해결할 수는 없다는 것을 알았다. 하드웨어의 제약, 시스템의 근본적인 문제는 더 고민해야 할 문제라는 것을 배웠다.
