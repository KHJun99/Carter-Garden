# 강지영(1423860) 공통 프로젝트 TIL
## 01/12 : 아이디어 회의
공통 프로젝트 주제 선정을 위한 아이디어 회의를 진행하였다.
여러 아이디어를 놓고 구현 난이도, 기술 수준, 프로젝트의 의미를 고려하여 주제를 논의하였다.

### 논의된 아이디어 정리
1. **무인 매장 운영 모니터링 시스템**  
    - 관리자 로봇을 통해 혼잡도와 장애물 파악 
    -> 아이디어가 약함
2. **스마트 카트**  
    - 자율주행 기능과 바코드 인식 기능 탑재 
    -> 난이도는 높지만 아이디어 측면에서는 가장 우수
3. **사무실 자세 교정 로봇**  
    - 거북목 등 잘못된 자세 교정 
    -> 사용자의 어떤 방향에서, 어떤 방식으로 자세를 인식할 것인지가 모호
4. **타자 연습 시스템**
    - 올바른 타자법을 손 모션 인식을 통해 게임처럼 학습하는 방식 
    -> 아이디어적으로 메리트가 없음
5. **게이밍 화분** 
    - 스마트 화분에 다마고치 형태의 게임 요소를 결합한 아이디어
    -> 게임 구현 자체의 난이도가 높고, AI를 쓰는 이유와 프로젝트의 의의가 명확하지 않음
6. **스마트 헬멧**   
    - 헬멧 착용자의 주변 장애물을 인식해 위험을 알려줌
    -> 고개 방향에 따른 자이로 제어와 인식 범위 설정이 어려움

### 느낀 점
아이디어 구상 시 
**“이게 실제로 가능한가”**, 그리고 **“그래서 어떤 불편함을 해결하는가”**를 설명하는 것이 중요하다는 걸 느꼈다.  
특히 AIoT 프로젝트에서는  
AI를 썼다는 사실보다 **AI가 없으면 해결하기 어려운 문제인지**가 핵심이라는 점이 인상 깊었다.

### 앞으로 더 생각해볼 것
- 스마트 카트 단일 구현으로 갈지, 더 큰 개념의 스마트 스토어로 확장할지
- 필요한 하드웨어와 기술 스택은 무엇인지
- 역할 분배를 어떤 방식으로 진행할지

위 사항을 고려하여,  
기획 단계에서 **“왜?”라는 질문이 생기지 않도록** 프로젝트를 구체화할 예정이다.

## 01/13 : 아이디어 구체화
전날 논의한 아이디어 중 **스마트 카트**를 중심으로,
단일 기능 구현이 아닌 **미래형 스마트 스토어 PoC**로 확장하는 방향을 구체화하였다.

이번 아이디어 구체화의 핵심은
스마트 쇼핑 카트를 중심으로 한 **사용자 경험 흐름을 스토리 형태로 정리하는 것**이었다.
단순히 카트가 이동하는 기능에 그치지 않고,
고객의 쇼핑 전 과정을 보조하는 자율주행 기반 서비스를 목표로 설정하였다.

### 사용자 시나리오
입장 → 상품 선택 → 자율주행 안내 → 상품 인식 → 결제 → 반납 또는 주차 안내

### MVP 기능 정리

- SLAM 기반 자율주행을 통한 상품 진열대 및 목적지 이동
- Wi-Fi 기반 웹 UI를 통한 상품 선택 및 쇼핑 흐름 제어
- NFC(RFID) 기반 상품 태깅을 활용한 장바구니 관리

4주라는 제한된 기간을 고려하여,
사람 추적과 같은 고난도 기능은 후순위로 두고
**안정적인 시연이 가능한 기능 위주로 MVP를 설정**하였다.
특히 상품 인식은 자동화를 무리하게 추구하기보다,
**의도 기반 태깅 방식**을 통해 인식 정확도와 구현 안정성을 우선시하기로 결정하였다.

### 느낀 점

구현해 보고 싶은 아이디어는 많았지만,
4주라는 제한된 기간과 사용 가능한 재료를 고려했을 때 현실적으로 구현이 어려운 아이디어가 많아 아쉬움이 남았다.
또한 아이디어를 실제로 구현한 이후,
**기존 시스템과 비교해 얼마나 효율적인지**,
그리고 **발표 과정에서 나올 수 있는 질문들에 충분히 대응할 수 있을지까지 함께 고려해야 한다**는 점이 생각보다 복잡하게 느껴졌다.
이를 통해 단순한 기능 구현을 넘어,
기획 단계에서부터 **타당성과 설명 가능성을 함께 고민하는 것이 중요**하다는 점을 깨달았다.

### 앞으로 할 일

- MVP 기준 기술 범위 및 구현 난이도 최종 확정
- 역할 분담 정리 및 핵심 기술에 대한 사전 학습
- 시연 환경을 고려한 데모 시나리오 구체화

## 01/14 : 환경 구축 및 방향 재정리

AIoT 키트 수령 후, 프로젝트 진행을 위한 **기본 환경 세팅**을 진행하였다.
키트 구성품 및 수량을 확인하고, 라즈베리파이에 **Raspberry Pi OS 설치 및 기본 설정**을 완료하였다.
네트워크 설정 후 **Flask 서버 구동 테스트**를 통해 웹 기반 제어 구조의 동작을 확인하였다.

또한 팀 미팅을 통해 시스템 구성을 재정리하였다.
구현 안정성과 일정 관리 측면을 고려하여
**스마트 카트는 1대만 사용**하고, **사람 추적 기능을 MVP에 포함**하는 방향으로 결정하였다.

이에 따라 단일 카트 기준의 사용자 상호작용과 핵심 기능 구현에 집중하기로 하였다.

### 느낀 점

아이디어 단계에서는 다양한 기능을 추가하고 싶었지만,
실제 하드웨어를 확인하고 환경을 직접 구성해 보니
**구현 범위를 명확히 제한하는 것이 오히려 프로젝트 완성도를 높일 수 있겠다는 생각이 들었다.**

### 앞으로 할 일

* 요구사항 명세서 작성
* 웹 UI 기본 화면 설계 및 구현
* 프로젝트에 필요한 센서 및 교보재  조사 후 주문

## 01/15 : 요구사항 명세서 작성 및 시뮬레이션 보드 구상

프로젝트의 전체 흐름을 정리하기 위해 **요구사항 명세서 작성**을 진행하였다.
또한 실제 시연을 고려하여 ***시뮬레이션 보드를 어떻게 구성할지 구상하고,
필요한 준비물들을 조사**하였다.

### 주요 진행 사항

### 1. 요구사항 명세서 작성

프로젝트에서 구현할 기능들을 막연한 아이디어 수준이 아니라,
**실제로 구현 가능한 단위**로 나누어 요구사항 명세서로 정리하였다.

하드웨어, 자율주행, 센서, 웹 기능 등을 구분하여
각 기능이 **필수인지(MVP)**, **후순위인지**를 명확히 구분하였다.

### 2. 시뮬레이션 보드 구성 방식 구상

실제 매장을 그대로 구현하는 것은 현실적으로 어렵기 때문에,
**소형 테스트 환경에서 시연 가능한 시뮬레이션 보드**를 구성하는 방향으로 논의하였다.

* 테스트 공간을 가정하고,
  진열대를 단순화한 구조물로 매장을 표현하는 방식 구상
* 실제 상품 대신 **RFID 태그가 부착된 상품 모형**을 활용하여
  기능 시연에 집중할 수 있도록 구성 계획 수립

### 3. 프로젝트 준비물 조사

시뮬레이션 보드와 초기 구현을 위해 필요한 준비물들을 정리하였다.

* RFID 리더기 및 태그
* 모터와 바퀴

필요한 물품을 미리 정리함으로써,
이후 구현 단계에서 불필요한 시간 낭비를 줄일 수 있을 것이라 생각하였다.

### 느낀 점

아이디어 단계에서는 기능 자체에만 집중했지만,
요구사항 명세서를 작성하고 시뮬레이션 환경을 구상하면서
**“어떻게 보여줄 것인가”와 “어디까지 구현할 것인가”가 훨씬 중요하다는 것을 느꼈다.**

특히 시연 환경을 고려하지 않은 기능은
아무리 좋아 보여도 실제 프로젝트에서는 의미가 없을 수 있다는 점이 인상 깊었다.
이번 정리를 통해 프로젝트를 **현실적인 범위 안에서 관리하는 것이 중요**하다는 것을 다시 한번 깨달았다.

### 앞으로 할 일

* 요구사항 명세서 기준 MVP 기능 최종 확정
* 준비물 목록 정리 후 구매 여부 결정
* 라즈베리파이 환경에서 GPIO와 Flask 연동

## 01/16 : 하드웨어 확정 및 사용자 경험(UX) 시각화

프로젝트의 실질적인 구현을 위해 핵심 부품을 주문하고, Figma를 통해 사용자가 스마트 카트를 이용할 때 마주할 인터페이스(UI)를 구체적으로 설계하였다.

### 주요 진행 사항

### 1. 하드웨어 사양 조사 및 부품 주문

RFID 인식 및 음성 안내 기능을 위해 리더기, 태그, 스피커의 사양을 상세히 검토하였다.

* **RFID 모듈 선정:** 라즈베리 파이 5와의 호환성을 최우선으로 고려하였다. 특히 리더기와 태그 간의 **주파수 매칭(13.56MHz 등)**이 필수적이기에 이를 확인하고, 배송 기간과 예산을 고려하여 **RC522 기반 모듈**을 최종 주문하였다.
* **음성 출력 장치:** 상품 인식 및 안내 상황을 사용자에게 알리기 위한 소형 스피커를 선정하였다.

### 2. Figma를 활용한 UI/UX 설계

단순한 기능 구현을 넘어, 실제 매장에서 고객이 편리하게 사용할 수 있는 화면 흐름을 Figma로 작업하였다.

* **화면 흐름 설계:** '입장 및 본인 인증 → 상품 탐색(자율주행 요청) → 장바구니 담기(RFID 태깅) → 결제 및 퇴장'으로 이어지는 시나리오를 시각화하였다.
* **7인치 터치스크린 최적화:** 카트에 장착될 화면 크기를 고려하여 버튼의 크기와 배치, 상품 리스트의 가시성을 조정하였다. 특히 이동 중인 카트에서도 조작이 쉽도록 **직관적인 UI 구성**에 집중했다.

### 3. 기술 스택 연동 및 프론트엔드 환경 구축

라즈베리 파이 환경에서 웹 기반 UI를 효율적으로 구동하기 위한 구조를 설계하였다.

* **Vue.js + Flask 연동:** Vite로 빌드한 Vue.js 정적 파일을 Flask 서버에서 서빙하거나, 개발 환경에서 프록시 설정을 통해 API를 호출하는 방식을 검토하였다.
* **라즈베리 파이 5 최적화:** 최신 OS 환경에서 외부 라이브러리 설치 시 발생할 수 있는 가상환경(venv) 이슈를 사전에 파악하였다.

### 4. 기능 명세서 상세화

* 사용자 추적(YOLO), 자율주행(SLAM), 상품 관리(RFID) 기능이 유기적으로 연결되는 흐름을 정리하여 **최종 기능 명세서**를 작성하였다.

### 느낀 점

Figma로 와이어프레임을 직접 그려보니, 머릿속으로만 생각했을 때 놓쳤던 불편한 점들이 보이기 시작했다. 

또한 하드웨어를 주문하며 **주파수나 전압 규격** 하나하나를 따지는 과정이 번거롭기도 했지만, 이 과정이 정확해야 나중에 삽질을 줄일 수 있다는 점을 깊이 체감했다. AIoT 프로젝트는 **'소프트웨어의 논리'와 '하드웨어의 물리적 규격'이 만나는 지점**을 잘 관리하는 것이 핵심인 것 같다.

### 앞으로 할 일

* 주문한 하드웨어 수령 전, Figma 시안을 바탕으로 Vue.js 기본 컴포넌트 개발
* Flask에서 RFID 인식 신호를 받았을 때 프론트엔드로 전달하는 API 구조 설계

## 01/19 : RFID 하드웨어 연동 및 키오스크 UI 고도화

**RC522 RFID 리더기를 라즈베리 파이 5에 연결**하고, 이를 제어하기 위한 소프트웨어 환경 구축 및 키오스크 UI의 완성도를 높이는 작업을 진행하였다.

### 주요 진행 사항

### 1. 하드웨어 물리 결합 및 통신 환경 구축

라즈베리 파이 5에서 하드웨어를 안정적으로 구동하기 위한 설정을 수행하였다.

* **전원 및 안전 관리:** 전자 회로 쇼트 방지를 위해 반드시 전원을 차단한 상태에서 핀을 연결해야 함을 숙지하고, RC522 모듈의 VCC를 3.3V에 정확히 결합하였다.
* **라이브러리 호환성 해결:** 라즈베리 파이 5에서 기존 `RPi.GPIO`가 동작하지 않는 문제를 확인하고, 이를 지원하는 `rpi-lgpio` 라이브러리로 대체하여 GPIO 제어 권한을 확보하였다.

### 2. RFID 태깅 및 데이터 기록(Write) 기능 구현

실제 상품 스티커에 데이터를 쓰고 읽는 테스트를 성공적으로 마쳤다.

* **`rfid_test.py`**: RFID 카드를 태깅했을 때 고유 ID(UID)와 내부에 저장된 텍스트 데이터를 정상적으로 읽어오는 것을 확인하였다.
* **`rfid_write.py`**: 스티커의 고유 ID는 고정되어 있지만, 내부 저장 공간에는 프로젝트 목적에 맞는 **상품 정보(텍스트)를 자유롭게 기록하고 수정**할 수 있는 기능을 구현하였다.

### 3. 키오스크 UI 및 가상 키보드 고도화

7인치 LCD 터치스크린 환경에 최적화된 사용자 인터페이스를 구축하였다.

## 01/20 : AWS EC2 배포 및 핵심 백엔드 기능 구현

라즈베리 파이 내부에서만 동작하던 로직을 확장하여 **AWS 클라우드 환경에 백엔드 시스템을 구축**하고, 프로젝트의 핵심 도메인인 **카트 대여 및 쿠폰 관리** 기능을 구현하였다.

### 주요 진행 사항

### 1. AWS EC2 기반 인프라 구축

로컬 환경을 벗어나 어디서든 접속 가능한 서버 환경을 구축하였다.

* **EC2 및 DB 서버 배포:** AWS EC2 인스턴스에 Flask 서버를 올리고, MySQL 데이터베이스를 연동하였다. 외부 접속을 위한 보안 그룹(3306, 5000 포트 등) 설정과 환경 변수(.env) 관리를 통해 보안성을 고려하였다.

### 2. 카트 및 쿠폰 도메인 로직 구현

Flask의 레이어드 아키텍처(Model-Schema-Service-Route)를 적용하여 확장성 있는 백엔드를 구축하였다.

* **카트 대여(Rent) API:** 카트의 상태(`status`)를 관리하고, 사용자가 카트를 대여할 때 `user_id`를 매핑하는 비즈니스 로직을 구현하였다.
* **쿠폰 시스템:** 유저별 보유 쿠폰 조회 및 사용 기능을 구현하였다. 특히 `marshmallow`를 사용하여 복잡한 DB 객체를 JSON으로 직렬화하고, `Utils/time.py`를 통해 쿠폰 만료 여부를 체크하는 타임스탬프 로직을 통합하였다.

### 3. RFID 기반 장바구니 연동 로직 설계

라즈베리 파이에서 인식된 RFID 데이터를 웹 UI를 거쳐 백엔드와 연동하는 흐름을 설계하였다.

* **데이터 흐름:** `라즈베리 파이(Python)` → `웹소켓` → `Vue.js` → `EC2 Flask API`로 이어지는 파이프라인 코드를 작성하였다.
* 태그의 메모리 영역에 저장된 `product_id`를 읽어 즉시 장바구니에 담는 로직을 구현하였으며, 하드웨어와의 실제 통합 테스트를 앞두고 있다.

### 느낀 점

단순히 기능을 만드는 것보다 **"서버에 올리고 연결하는 과정"**이 생각보다 변수가 많다는 것을 배웠다. 로컬에서는 잘 돌아가던 코드가 서버 환경에서 DB 연결 문제로 막힐 때 당황하기도 했지만, 트레이스백을 따라가며 문제를 해결하는 과정에서 인프라에 대한 이해도가 높아졌다.

또한, 하드웨어(RFID)에서 보낸 데이터가 클라우드 서버(EC2)까지 전달되어 장바구니에 담기는 전체 흐름을 코드로 구현해보니, 프로젝트의 뼈대가 드디어 완성되어가고 있다는 확신이 들었다.

### 앞으로 할 일 (내일 계획)

* **하드웨어 통합 테스트:** 오늘 짠 RFID 인식 및 장바구니 추가 코드 실제 동작 확인
* **UX 요소 추가:** 상품 인식 성공 시 스피커를 통해 알림음(Beep)이 출력되는 기능 구현
* **중간 발표 준비:** 현재까지의 진행 상황 및 기술 스택 정리

* **장바구니 삭제 다이얼로그 UI 변경:** 상품 삭제 시 나타나는 안내 창의 가시성을 높여 사용자의 오작동을 방지하도록 디자인을 고도화하였다.

## 01/23 : WSL2 기반 ROS2 시뮬레이션 환경 구축 및 중간 발표

**WSL2에 ROS2 Humble을 설치**하고, Gazebo 시뮬레이션을 위한 **GPU 최적화**를 진행하였다. 또한, 프로젝트 **중간 발표**를 통해 객관적인 피드백을 수령하였다.

### 주요 진행 사항

### 1. WSL2 및 ROS2 Humble 환경 구축

가상머신 대비 리소스 소모가 적은 WSL2 환경에서 ROS2 개발 환경을 구축하였다.

* **ROS2 Humble 설치:** GPG 키 등록 및 저장소 설정을 거쳐 `desktop-full` 버전을 설치하고, `colcon`, `rosdep` 등 빌드 및 의존성 관리 도구 설정을 완료하였다.
* **환경 변수 자동화:** 터미널 실행 시마다 설정을 반복하지 않도록 `.bashrc`에 `setup.bash` 및 그래픽 설정을 추가하였다.

### 2. GPU(RTX 4050) 인식 및 Gazebo 연동 최적화

시뮬레이션의 실시간 렌더링 성능을 확보하기 위해 하드웨어 가속 설정을 진행하였다.

* **GPU 우선순위 설정:** WSL2가 내장 그래픽 대신 **NVIDIA RTX 4050**을 사용하도록 윈도우 그래픽 설정 및 NVIDIA 제어판을 조정하고, `MESA_D3D12_DEFAULT_ADAPTER_NAME` 환경 변수를 통해 가속 어댑터를 강제 지정하였다.
* **Gazebo 실행 및 오류 해결:** 로봇 모델 로딩 시 발생하는 타임아웃 문제를 해결하기 위해 `launch_sim.launch.py` 코드 내 `spawn_entity` 노드에 `timeout` 파라미터(120s)를 추가하고 재빌드하였다.

### 3. 중간 발표 진행 및 피드백 수렴

현재까지의 진행 상황을 발표하고 컨설턴트 및 코치님의 피드백을 통해 보완점을 파악하였다.

* **컨설턴트님의 피드백:** 제품의 가치를 어필할 수 있는 핵심 요소에 집중할 것, 하드웨어 선정(모터 등) 시 성능 수치와 무게 등을 고려한 '이유'를 명확히 설명할 것, 비전문가도 이해하기 쉬운 '사용자 플로우' 중심의 스토리텔링 필요.
* **코치님의 피드백:** 발표 시간(8분 50초)과 PPT 구성, 비용 데이터 활용은 긍정적이나, 설계 과정의 문제 해결 사례 보강 필요, 이미지 비율 및 애니메이션 프레임 등 시각적 디테일 개선, 최종 발표 전 실제 환경 테스트 및 시연 위주의 구성 추천.

---

### 느낀 점

WSL2 환경에서 GPU 가속을 설정하며 윈도우와 리눅스 사이의 하드웨어 자원 할당 과정을 깊이 있게 이해할 수 있었다. 특히 중간 발표 피드백을 통해 **"기술적으로 무엇을 만들었는가"** 만큼이나 **"왜 이 기술을 선택했고, 사용자의 어떤 불편함을 해결하는가"**를 논리적으로 전달하는 것이 프로젝트의 퀄리티를 증명하는 핵심임을 깨달았다.

### 앞으로 할 일

## 01/26 : 하드웨어 환경 변경 및 지능형 추종 기능 시뮬레이션

구동 환경을 라즈베리 파이 5에서 **라즈베리 파이 4**로 변경하고 기존 기능과의 호환성을 검증하였으며, Gazebo 환경에서 **객체 인식 기반의 지능형 추종 기능** 구현에 집중하였다.

### 주요 진행 사항

### 1. 하드웨어 구동 환경 변경 및 시스템 셋업

프로젝트의 안정성과 전력 효율을 고려하여 제어 보드를 라즈베리 파이 5에서 4로 변경하고 환경을 재구축하였다.

* **OS 재구축:** 라즈베리 파이 4 규격에 맞는 OS 이미지를 새로 굽고, 프로젝트 구동에 필요한 기본 라이브러리 및 환경 설정을 처음부터 재진행하였다.
* **RFID 및 키오스크 호환성 검증:** 보드 변경 후 기존에 구현했던 **RFID 연동 로직과 키오스크(Vue.js) UI**가 정상적으로 통신하는지 확인하였다

### 2. 지능형 객체 인식 및 추종 기능 구현 (미완성)

Gazebo 시뮬레이션 환경에서 카메라와 라이다(LiDAR)를 융합하여 특정 대상을 인식하고 따라가는 로직을 고도화하고 있다.

* **YOLO 기반 객체 인식 연동:** 카메라 노드를 통해 들어오는 영상 데이터에서 YOLO를 활용해 사람 또는 특정 사물을 식별하는 인식 레이어를 구성하였다.
* **추종(Follower) 노드 설계:** 인식된 객체의 좌표값을 기반으로 로봇의 선속도와 각속도를 제어하여 대상을 일정 거리에서 따라가는 알고리즘을 구현하였다.

* **현재 진행 상황:** 기본 인식 및 추격 로직은 작성되었으나, 시뮬레이션 환경에서의 부드러운 움직임과 타겟 실종 시 예외 처리 등 고도화 작업이 진행 중이다.

### 3. 라이브러리 의존성 문제 해결

다양한 인공지능 모델(TorchReid, InsightFace 등)을 ROS2 노드에 통합하는 과정에서 발생하는 환경 충돌을 해결하였다.

* **NumPy 및 의존성 정비:** `cv_bridge`와 AI 라이브러리 간의 NumPy 버전 충돌 이슈를 버전 다운그레이드를 통해 해결하며 개발 환경을 정상화하였다.

## 01/27 : 객체 인식 기반 지능형 추종 알고리즘 고도화

시뮬레이션 환경에서 카메라(YOLO)와 라이다(LiDAR) 센서를 융합하여 주인을 안정적으로 따라가는 **지능형 추종(Follower) 노드**를 구현하였다.

### 주요 진행 사항

### 1. 센서 퓨전 기반 추종 로직 구현

단순히 카메라만 사용하는 방식에서 벗어나, 카메라로 방향을 잡고 라이다로 정확한 거리를 측정하는 하이브리드 방식을 적용하였다.

* **주인 식별 시스템:** YOLO 트래킹 ID가 1000번 이상인 객체만을 주인으로 인식하여 추종을 시작하도록 설계하였다.
* **라이다 정밀 보정:** 카메라 화각 내에서 가장 가까운 물체를 라이다로 탐색하여 `target_angle`을 미세 조정함으로써 정밀도를 높였다.

### 2. 제어 알고리즘 및 예외 처리

* **P-제어 적용:** 주인과의 거리 오차(`dist_error`)에 비례하여 속도를 조절하는 로직을 통해 부드러운 가속과 감속을 구현하였다.
* **기억 유지(Timeout) 기능:** 주인을 일시적으로 놓치더라도 3초간 마지막 위치를 기억하고 추적을 시도하는 예외 처리 로직을 추가하였다.

### 느낀 점

인식된 데이터의 좌표값을 실제 로봇의 움직임(`Twist`)으로 변환하는 과정에서 부호 계산과 필터링의 중요성을 깨달았다. 특히 시뮬레이션 환경에서의 물리적 저항을 고려한 최소 속도 설정이 실제 주행 안정성에 큰 영향을 미친다는 점이 인상 깊었다.

## 01/28 : Nav2 하이브리드 제어 및 웹 키오스크 고도화

단순 추종의 한계를 극복하기 위해 **Nav2와 직접 제어를 결합한 하이브리드 시스템**을 구축하고, 서비스 완성도를 위해 **웹 키오스크의 결제 및 데이터 관리 로직**을 강화하였다.

### 주요 진행 사항

### 1. 하이브리드 추종 시스템(Nav2 + Direct Control) 구축

좁은 모퉁이나 장애물이 많은 환경에서도 충돌 없이 주인을 따라갈 수 있도록 제어 방식을 이원화하였다.

* **경로 안전 검사(`check_path_safety`):** 로봇 전방 30cm 이내 장애물 유무와 사람이 있는 방향으로의 길목이 확보되었는지 실시간으로 검사한다.
* **동적 모드 전환:** 경로가 안전할 때는 반응성이 빠른 직접 제어(`drive_direct`)를 사용하고, 장애물이나 코너가 감지되면 Nav2(`drive_nav2`)가 경로를 재계산하여 회피하도록 설계하였다.

### 2. 웹 키오스크 비즈니스 로직 및 DB 강화

사용자가 실제 쇼핑을 마무리하는 과정을 고려하여 백엔드와 UI를 보완하였다.

* **결제 시스템 로직 설계:** 장바구니에 담긴 상품 리스트를 기반으로 최종 금액을 산출하고 결제 프로세스로 이어지는 화면 흐름을 구현하였다.
* **사용자 세션 관리:** 회원 및 비회원 로그인 시 이전 사용자의 장바구니 데이터가 남지 않도록 **장바구니 자동 초기화 로직**을 추가하여 데이터 무결성을 확보하였다.
* **DB 스키마 확장:** 결제 이력 및 사용자별 맞춤 정보를 관리하기 위한 테이블 구조를 추가 설계하였다.

### 느낀 점

Nav2의 Costmap을 활용하니 직접 코드로 장애물 회피 로직을 짤 때보다 훨씬 안정적인 움직임이 가능했다. 또한, 기술적 구현만큼이나 중요한 것은 **"데이터의 흐름이 끊기지 않는 것"**이며, 로그인부터 결제까지의 예외 상황을 처리하는 과정이 서비스의 신뢰도를 결정한다는 것을 다시 한번 실감했다.

### 앞으로 할 일

* 실기기(Jetson Orin Nano) 이식을 위한 하드웨어 성능 최적화
* 전체 시나리오(입장-쇼핑-결제-반납) 통합 테스트 및 시연 영상 촬영 준비

## 01/30 : 시스템 최적화 및 추종 알고리즘 예외 처리 시도

프로젝트의 실질적인 구동 안정성을 높이기 위해 불필요한 로그 데이터를 정리하고, 키오스크의 입력 편의성과 피드백 UI를 개선하였다. 또한, 가제보(Gazebo) 시뮬레이션 환경에서 발생하는 추종 시스템의 고질적인 문제를 해결하기 위한 고도화 작업을 진행하였다.

### 주요 진행 사항

### 1. 추종(Follower) 기능 실행 스크립트 작성 및 테스트

추종 노드를 개별적으로 실행하던 번거로움을 줄이기 위해 **실행 스크립트를 작성**하여 개발 효율을 높였다. 가제보 환경에서 터틀봇이 사람을 인식하고 따라가는 일련의 과정을 자동화하여 반복 테스트가 가능하도록 구성하였다.

### 2. 추종 알고리즘 예외 처리 시도 (미해결)

시뮬레이션 중 터틀봇이 추적 대상을 놓치면 그 자리에 멈춰버리는 현상을 해결하기 위해 로직 수정을 시도하였다.

* **문제점:** 사람이 급격하게 방향을 틀거나 장애물 뒤로 숨을 경우 인식 범위를 벗어나 시스템이 '정지' 상태로 고착됨.
* **시도 내용:** 대상의 마지막 위치 정보를 활용해 해당 방향으로 저속 이동하며 재탐색을 시도하는 로직을 적용해보았으나, 시뮬레이션 환경의 변수로 인해 완벽한 해결에는 도달하지 못해 추가적인 보완이 필요한 상태이다.

### 3. 키오스크 UI/UX 및 시스템 최적화

사용자 인터페이스의 디테일을 다듬어 서비스 완성도를 높였다.

* **한글 입력 로직 수정:** 가상 키보드 사용 시 자음과 모음이 분리되거나 오입력되는 현상을 수정하여 입력 안정성을 확보하였다.
* **장바구니 스낵바(Snackbar) 도입:** 상품이 장바구니에 정상적으로 담겼을 때 화면 하단에 알림 메시지(스낵바)를 띄워 사용자가 동작 성공 여부를 즉각 인지할 수 있도록 개선하였다.

### 느낀 점

추종 알고리즘에서 타겟을 상실했을 때의 예외 처리가 생각보다 까다롭다는 것을 느꼈다. 비록 오늘 완벽하게 해결하지는 못했지만, 어떤 지점에서 로봇이 멈추는지 원인을 파악한 것에 의의를 두고 싶다. 또한, 스낵바와 같은 작은 UI 요소 하나가 전체적인 사용성을 크게 개선한다는 점을 다시 한번 확인했다.

* 하드웨어 완성 및 전체 시스템 통합

## 02/02 : 추종 모드 실행 자동화 및 정밀 매핑

개선된 추적 알고리즘을 즉시 테스트할 수 있는 환경을 구축하고, 시뮬레이션 공간의 물리적 수치를 측정하여 데이터화하였다.

### 주요 진행 사항

* **추종 모드 통합 실행 스크립트 작성**
* 가제보 실행, 타겟(small_human) 소환, Nav2 및 상태 브로드캐스터 실행을 한 번에 처리하는 **Bash 스크립트(`run_follower.sh`)**를 작성하였다.
* `sleep`을 적절히 배치하여 노드 간 로드 순서를 보장함으로써 실행 시 발생하는 충돌을 방지하였다.


* **공간 측정 및 직접 매핑**
* 시뮬레이션 공간의 크기를 정밀하게 측정하고 이를 바탕으로 직접 맵을 제작하였다. 이는 로봇이 자신의 위치를 파악(Localization)하는 데 있어 오차를 줄이는 중요한 기초 작업이 되었다.

### 느낀 점

스크립트 자동화를 통해 매번 터미널을 여러 개 띄워야 했던 번거로움을 줄이니 테스트 효율이 눈에 띄게 좋아졌다. 또한, 단순히 자동 생성된 맵을 쓰는 것보다 실제 수치를 측정해 매핑하는 과정에서 **가상 환경과 물리적 수치의 일치성**이 얼마나 중요한지 깨달았다.

### 앞으로 할 일

* 실제 매장과 유사한 장애물 배치 후 주행 안정성 테스트
* YOLO 인식 노드와 PID 추종 노드 간의 데이터 병목 현상 확인

## 02/03 : 시뮬레이션 환경 고도화 및 맵 리뉴얼

시연 환경의 현실감을 높이기 위해 실제 장애물을 배치하고, 변화된 환경에 맞춰 맵 데이터를 갱신하였다.

### 주요 진행 사항

* **동적 환경 구성 및 장애물 배치**
* 시뮬레이션 공간에 실제 매장에서 마주할 수 있는 다양한 장애물들을 배치하였다. 이 과정에서 통로 간격 등이 변경됨에 따라 로봇의 이동 경로 최적화가 필요해졌다.


* **3D 및 2D 맵 리뉴얼 (Mapping)**
* 장애물이 추가된 환경을 다시 스캔하여 **3D 맵과 2D 점유 격자 지도(Occupancy Grid Map)**를 새롭게 생성하였다.
* 좁아진 간격에서도 Nav2가 원활하게 경로를 생성할 수 있도록 환경 설정을 세밀하게 조정하였다.

### 느낀 점

장애물이 늘어날수록 로봇이 갈 수 있는 '유효 경로'가 좁아진다는 것을 맵 데이터를 통해 시각적으로 확인했다. 환경이 변하면 기존의 맵은 무용지물이 될 수 있다는 점을 실감하며, **환경 변화에 유연하게 대응하는 매핑 기술**의 중요성을 느꼈다.

### 앞으로 할 일

* 좁은 공간 통과를 위한 안전 거리 파라미터 최적화
* Jetson Orin Nano 실기기 이식 및 주행 동작 확인

## 02/04 : 하드웨어 트러블슈팅 및 센서 퓨전 직진 제어

이때까지 발생했던 하드웨어 문제의 근본 원인을 해결하고, 여러 센서 데이터를 융합하여 로봇의 물리적 주행 성능을 비약적으로 향상시켰다.

### 주요 진행 사항

* **하드웨어 결함 해결: 브레드보드 교체**
* 인코더 값이 부정확하게 들어오던 고질적인 문제를 추적한 결과, **브레드보드의 접촉 불량**이 원인임을 밝혀냈다. 부품 교체 후 모터 데이터가 정상 수집되는 것을 확인하였다.


* **센서 퓨전 기반 직진 제어 구현**
* LiDAR, IMU, Odometry 데이터를 결합한 `StraightFusionController`를 구현하였다.
* IMU/Odom의 Yaw 값을 퓨전하여 방향을 보정하고, LiDAR로 좌우 벽면과의 거리를 계산해 중앙 주행을 유지하는 로직을 통합하였다.


* **발표 준비 시작**
* 최종 발표를 위한 PPT 표지와 기본 레이아웃 제작을 완료하였다.

### 느낀 점

하드웨어 프로젝트에서 "왜 안 되지?" 싶은 문제는 결국 아주 기초적인 물리 연결(브레드보드)에서 올 수 있다는 것을 배웠다. 또한, 단일 센서만으로는 불가능했던 정교한 직진 주행이 **센서 퓨전(Fusion)**을 통해 안정화될 수 있다는 것을 알게 되었다.

### 앞으로 할 일

* Jetson Orin Nano 제어 노드 최적화를 통한 실시간 주행 안정성 확보
* 시뮬레이션과 실기기 간의 간극 조정을 위한 전체 시나리오 통합 검증
