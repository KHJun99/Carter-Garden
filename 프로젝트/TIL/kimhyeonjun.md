- 사용자 여정 8단계 MVP 시나리오 설계

## 배운 점

### MVP 시나리오 설계 방법
**전체 사용자 여정을 8단계로 구조화**:
1. 고객 입장 → 카트 마중
2. 웹 UI에서 상품 선택
3. 카트 자율 주행
4. NFC 태깅 → 장바구니 추가
5. 상품 제거 기능
6. 결제 프로세스
7. 이동 지원 (출구/주차장)
8. 카트 자동 반납

### 시스템 설계 원칙
- **기능 분리**: 웹(UI, 결제, 상품 관리) vs 하드웨어(자율주행, 센서, NFC)
- **인터페이스 정의**: 웹-하드웨어 간 통신 프로토콜 명확화
- **사용자 중심 설계**: 고객의 쇼핑 여정 전체를 고려한 시나리오

### 협업 프로세스
- 6명 팀원의 강점을 고려한 역할 분담
- 하드웨어-소프트웨어 통합 지점 명확화
- 주간 마일스톤 설정

## 어려웠던 점 & 해결

### 문제: 기능 간 의존성 관리
**어려웠던 점**:  
자율주행, NFC 태깅, 결제 시스템이 서로 연동되어야 하는데 개발 순서 결정이 어려움

**해결**:
- 핵심 기능부터 순차적 개발
- 모의 데이터(Mock Data)로 독립 개발 가능하도록 설계
- API 인터페이스 먼저 정의 후 병렬 개발

**교훈**:  
복잡한 시스템은 느슨한 결합(Loose Coupling)과 명확한 인터페이스 정의가 핵심

## 느낀점
- 어제의 추상적인 아이디어가 오늘 구체적인 8단계 시나리오로 정리되니 프로젝트가 현실감 있게 다가왔다.

- SSAFY에서 배운 애자일 방법론이 하드웨어 프로젝트에도 적용 가능하다는 것을 확인했다.

# 2026-01-16 TIL : Jetson Orin Nano 전원 시스템 설계

## 오늘 한 일
- L298N 모터 드라이브와 JGA25-370 DC 모터 2개를 이용한 RC카 제작 진행
- Jetson Orin Nano GPIO 핀 모터 제어 방법 학습
- 18650 배터리 홀더 활용한 12V 전원 공급 시스템 구성
- 전원 공급 회로 설계의 핵심 발견: Jetson은 5V 입력 사용 (19V 아님!)
- DC-DC 벅 컨버터 필요성 인식
- 안전한 배터리 충전 시스템 이해

## 배운 점

### 전원 관리의 중요성
**컴포넌트별 전압 요구사항**:
- 모터: 12V (18650 3직렬 배터리)
- Jetson Orin Nano: **5V** (중요! 19V 아님)
- DC-DC 벅 컨버터: 12V → 5V 변환 필수

**전원 시스템 구성**:
```
18650 배터리 (3직렬) = 12V
    ↓
├─→ L298N 모터 드라이버 (12V) → DC 모터 2개
│
└─→ DC-DC 벅 컨버터 (12V → 5V) → Jetson Orin Nano (5V)
```

### 배터리 시스템 구성
- **3S LiPo (11.1V)** vs **18650 3직렬 (12V)** 비교
- **충전 회로와 방전 회로 분리** 필요성
- **과방전 보호** 기능의 중요성
- **안전 충전**: 충전 시 부하(모터, Jetson) 연결 해제 필수
- 적정 충전 전압: 12.6V

### 모터 제어 회로
- L298N 모터 드라이버: 외부 전원 공급 필수
- GPIO를 통한 PWM 신호로 모터 속도 제어
- **모터 전류는 외부 배터리에서 공급, Jetson은 신호만 제공**

## 어려웠던 점 & 해결

### 문제 1: 전원 시스템 구성의 복잡성
**어려웠던 점**:  
여러 전압 레벨(12V, 5V)을 동시에 관리해야 하는 복잡성

**해결**:
- DC-DC 벅 컨버터 도입으로 12V → 5V 안정적 변환
- 전원 공급 회로 다이어그램 작성
- 각 컴포넌트별 전압/전류 요구사항 문서화

**교훈**:  
**하드웨어 프로젝트에서는 전원 설계가 가장 기초이자 핵심**

### 문제 2: 참고 자료의 불완전성
**어려웠던 점**:  
튜토리얼 영상에서 DC-DC 벅 컨버터 설치 과정이 생략됨

**해결**:
- 직접 Jetson Orin Nano 박스에서 스펙 확인 (5V 입력)
- 논리적 추론으로 필요 부품(DC-DC 벅 컨버터) 파악

**교훈**:  
**"가정하지 말고, 확인하라"** - 튜토리얼을 맹신하지 않고 항상 공식 스펙 문서 확인 필수

- 오늘 가장 중요한 교훈은 **"가정하지 말고, 확인하라"**였다. 처음에는 Jetson이 19V를 사용할 것이라고 막연히 생각했지만, 실제 박스를 확인하니 5V였다. 만약 확인하지 않고 진행했다면 부품을 잘못 구매하거나 **최악의 경우 Jetson을 손상**시킬 수도 있었을 것이다.

- 로봇공학 전공 배경이 있지만, **전원 관리나 임베디드 시스템 통합은 여전히 새로운 영역**이다. SSAFY에서 소프트웨어를 배우면서 하드웨어와 소프트웨어를 통합하는 능력이 점점 향상되고 있음을 느낀다. 오늘의 시행착오가 내일의 자산이 될 것이라 믿는다.

### 1. 전력 시스템의 기본 원리
```python
# 전력 보존 법칙
P = V × I (전력 = 전압 × 전류)

# Buck 컨버터 예시
입력: 20V × 1.5A = 30W
효율: 90%
출력: 5V × 5A = 25W
→ 30W × 0.9 = 27W > 25W ✓ 가능!
```

**핵심 깨달음**: 5V/3A를 5V/5A로 직접 변환은 불가능하지만, 높은 전압(20V)에서 시작하면 Buck 컨버터로 5V/5A 변환 가능! 전압을 낮추면 전류가 높아지는 원리 활용.

### 2. Buck vs Boost 컨버터의 차이

| 구분 | Buck (강압) | Boost (승압) |
|------|------------|-------------|
| 전압 | 낮춤 (12V→5V) | 높임 (3.7V→5V) |
| 전류 | 높아짐 | 낮아짐 |
| 효율 | 85~95% | 70~85% |
| 용도 | 높은 전압→낮은 전압 | 낮은 전압→높은 전압 |

### 문제 1: "5V/3A를 5V/5A로 변환할 수 없다는 사실"
**어려웠던 점**:  
처음에는 Buck 컨버터만 있으면 5V/3A를 5V/5A로 변환할 수 있을 줄 알았음. 하지만 전력 보존 법칙 때문에 불가능.

**해결**:  
- 전력(W) = 전압(V) × 전류(A) 공식 정확히 이해
- Buck 컨버터는 **전압을 낮추고 전류를 높이는** 장치
- 5V → 5V 변환은 의미 없음, 더 높은 전압(20V)에서 시작해야 함
```
20V/3A → Buck → 5V/5A
입력: 60W → 출력: 25W ✓ 가능!
```

### 문제 4: USB-C PD 프로파일 선택의 혼란
**어려웠던 점**:  
보조배터리가 여러 전압(5V, 9V, 12V, 15V, 20V)을 지원하는데, 어떤 걸 선택해야 5V/5A를 만들 수 있는지 혼란.

**해결**:  
- **높은 전압 선택**이 핵심 (20V 또는 15V)
- 5V/3A 선택 → 15W (부족)
- 20V/5A 선택 → 100W → Buck 컨버터 → 5V/5A (충분!)
- PD 트리거 모듈로 원하는 전압 강제 선택 가능

## 느낀점

### 1. "안 된다"는 것과 "방법을 모른다"는 것의 차이
처음에는 내가 가진 5V/3A 보조배터리로는 라즈베리파이 5를 구동할 수 없다고 생각했다. 하지만 USB-C PD의 높은 전압 프로파일(20V)을 활용하면 충분히 가능하다는 것을 알게 되었다. 문제를 다른 각도에서 접근하는 것의 중요성을 느꼈다.

### 2. 기초 원리의 중요성
전력 = 전압 × 전류라는 간단한 공식이 전체 시스템 설계의 기반이 된다. 단순히 부품을 연결하는 것이 아니라, 물리 법칙을 이해하고 적용하는 것이 엔지니어링의 본질임을 깨달았다.

### 3. 하드웨어 디버깅은 소프트웨어와 다르다
LiDAR 에러 해결 과정에서 로그만 보고 소프트웨어 설정만 바꾸는 것은 한계가 있었다. Python으로 직접 하드웨어와 통신하며 "어느 단계에서 문제인지" 명확히 파악하는 것이 훨씬 효과적이었다. 앞으로는 하드웨어 문제도 직접 테스트하는 습관을 들여야겠다.

### 4. 안전은 선택이 아닌 필수
리튬이온 배터리를 다루면서 잘못하면 화재나 폭발이 발생할 수 있다는 것을 알게 되었다. BMS, 퓨즈, 극성 확인 등이 귀찮은 작업이 아니라 생명과 직결된 문제라는 것을 절감했다. 빠른 개발보다 안전한 개발이 우선이다.

### 5. SSAFY에서 비전공자로서의 성장
전기/전자 기초 지식이 부족했지만, 오늘 하루 동안 Buck 컨버터, PD 프로토콜, 배터리 관리 시스템 등을 배우면서 많은 것을 흡수했다. 비전공자라는 핸디캡은 맞지만, 배우려는 의지와 올바른 방향성만 있으면 충분히 따라잡을 수 있다는 자신감이 생겼다.

- 문제 원인 파악: 납땜 과정에서 손상된 빵판
- 빵판 교체 후 엔코더 정상 작동 확인 및 테스트 완료

## 배운 점

### 하드웨어 구성 확인
- JGA25-371 모터 엔코더 핀 구성 (6핀)
  - 핀 1: Motor - (빨강)
  - 핀 2: Hall GND (검정)
  - 핀 3: Hall B (노랑)
  - 핀 4: Hall A (녹색)
  - 핀 5: Hall 3.3V (파랑)
  - 핀 6: Motor + (흰색)
- 엔코더는 3.3V로 작동하며, Jetson GPIO에서 병렬로 2개에 전원 공급 중

### 체계적인 디버깅 순서의 중요성
1. 전원 용량 확인 (60W 보조배터리 - 충분)
2. 공통 접지 확인 (Motor HAT, Jetson 모두 연결됨)
3. 회로 구성 확인 (올바른 연결)
4. **물리적 연결 상태 확인** ← 여기서 문제 발견!

### 빵판(Breadboard) 사용 시 주의사항
- 빵판은 납땜용이 아닌 임시 프로토타이핑용
- 고온 납땜 시 플라스틱 변형 및 내부 스프링 접점 손상 가능
- 납땜 열로 인한 손상 → 내부 접촉 불량 → 저항 증가 → 전압 강하 발생

## 어려웠던 점 & 해결

**증상:**

- 전압 강하 발생

- Jetson GPIO 3.3V 핀의 전류 공급 능력 부족

**실제 원인:**
- 빵판에 납땜하는 과정에서 열로 인한 빵판 손상
- 내부 접촉 불량으로 저항 증가 → 전압 강하

**해결:**
- 새 빵판으로 교체 → 정상 작동 확인

## 배운 점

### ROS2 Launch 파일 실행 순서 관리
- Launch 파일에서 노드들의 실행 순서가 중요하며, 특히 Gazebo 같은 시뮬레이터는 완전히 로딩될 때까지 기다려야 함
- `ExecuteProcess`보다 `IncludeLaunchDescription`이 Nav2 같은 복잡한 시스템 실행에 더 안정적
- 이벤트 핸들러를 사용한 순차 실행 로직 구현 방법

### Jetson 환경의 특수성
- Jetson 보드는 일반 PC와 달리 NVIDIA가 제공하는 전용 PyTorch 빌드를 사용해야 함
- ARM 아키텍처 특성상 pip로 일반 PyTorch를 설치할 수 없음
- NumPy 버전 호환성이 ROS2 패키지(특히 cv_bridge)와 충돌할 수 있음

### 의존성 관리의 중요성
- 패키지 간 버전 호환성 문제는 사전에 확인하고 관리해야 함
- NumPy 2.x는 많은 기존 라이브러리와 호환되지 않아 1.x 버전 유지 필요
- `numpy<2.0` 같은 버전 제약 조건을 requirements.txt에 명시하는 습관 필요

### ROS2 토픽 통신 디버깅 절차

1. **TensorRT 변환**: YOLO 모델을 TensorRT 엔진으로 변환 시 메모리 40% 감소, 속도 2-3배 향상

   - 30프레임마다 캐시 정리
   - `torch.cuda.empty_cache()` 주기적 실행

### 문제 3: 메모리 모니터링 필요성
**해결**:
```python
# 메모리 모니터링 타이머 추가
def check_memory(self):
    if self.device.type == 'cuda':
        allocated = torch.cuda.memory_allocated() / (1024**3)
        total = torch.cuda.get_device_properties(0).total_memory / (1024**3)
        gpu_usage = (allocated / total) * 100
        
        if gpu_usage > 90:
            self.force_cleanup()

# 강제 정리 메서드
def force_cleanup(self):
    self.cached_results.clear()
    if len(self.owner_gallery) > 15:
        self.owner_gallery = self.owner_gallery[-15:]
    gc.collect()
    torch.cuda.empty_cache()
```
