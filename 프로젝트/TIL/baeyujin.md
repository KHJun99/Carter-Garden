  - 키보드 입력, 서보 각도 제어, DC 모터 속도/방향 제어 흐름을 정리하고, 오린카 주행·조향에 연결될 수 있도록 코드 구조를 맞추는 작업을 진행함.
  - 오린카에 들어갈 서보 모터, DC 모터, PCA9685, Jetson Orin, RFID 리더 등 주요 부품을 리스트업하고, 각 부품의 역할과 연결 방식을 문서로 정리함.


- 네트워크 / 개발 환경 
  - Jetson 보드에 SSH로 원격 접속하는 방법을 학습하고, 실제로 연결 테스트를 진행함. 
  - 처음에는 WSL(Ubuntu) 환경에서 `ssh <user>@<JETSON_IP>` 형태로 접속을 시도했으나, 연결 문제로 인해 정상 접속이 되지 않는 이슈를 경험함. 
  - 환경을 Windows PowerShell로 바꿔 동일한 SSH 접속을 시도했고, Jetson Orin에 원격 접속하는 데 성공함.

- Git / 협업 환경
  - WSL 환경에서 Git을 활용해 로컬 작업 내용을 GitHub 원격 저장소에 올리는 전체 플로우(`git init`/`clone` → `add` → `commit` → `push`)를 직접 수행해 봄.
  - WSL에서의 Git 사용 시 원격 저장소 URL, 인증 방식(HTTPS/SSH) 설정이 중요하다는 것을 확인함. 

- 기술·부품 및 구조 정리 
  - 현재까지 사용 중이거나 사용할 예정인 기술 스택(예: Jetson Orin 기반 제어, PCA9685 PWM 제어, SSH 원격 개발, WSL + Git 등)을 한 번에 정리하는 시간을 가짐. 
  - 각 기술과 부품이 프로젝트 전체 흐름에서 어디에 쓰이는지, 어떤 의존성이 있는지 개략적인 구조를 메모로 정리함. 


## 배운 점  
- Jetson처럼 임베디드 보드를 쓸 때는, 물리적인 접근 없이도 SSH로 원격 제어 가능한 환경을 만들어 두면 개발 속도와 디버깅 효율이 크게 향상된다는 것을 체감함.
- 같은 SSH 명령이라도 WSL과 Windows PowerShell처럼 환경이 다르면 방화벽, 네트워크 설정, 인증 방식 차이로 인해 동작이 달라질 수 있어, 문제 발생 시 “환경을 바꿔 보는 것”이 중요한 해결 전략이라는 걸 배움. 
- WSL에서 Git을 사용하는 법을 익히면서, 로컬 개발 환경을 어떻게 구성하느냐에 따라 협업 방식과 배포 플로우가 달라진다는 점을 이해함. 
- 하드웨어 부품과 기술 스택을 한 번에 정리해 보니, 앞으로 어떤 부분을 먼저 구현하고 어떤 부분은 나중에 붙일지 우선순위를 정하기가 훨씬 쉬워졌다고 느꼈음.


## 느낀 점  
- SSH, Git 같은 개발 환경 문제를 직접 부딪혀서 해결해 보니, 단순 이론 공부보다 실전으로 한 번 해본 경험이 훨씬 오래 기억에 남는다는 느낌을 받음.
- Jetson 원격 접속 문제를 WSL에서 PowerShell로 바꿔 해결한 경험 덕분에, 앞으로도 비슷한 네트워크/환경 이슈가 나와도 겁먹지 않고 여러 방법을 시도해 볼 자신감이 생김. 
- 오늘 정리해 둔 부품·기술 구조가 나중에 팀원들과 역할을 나누거나 설계를 바꿔야 할 때 기준점이 될 것 같아, “정리하는 시간도 개발의 일부”라는 생각이 들었음.

## 배운 점

### [ROS2 노드 기본 구조 이해]
- rclpy.init() → Node 생성 → rclpy.spin(node) → Ctrl+C 종료로 이어지는 ROS2 Python 노드의 기본 패턴이 모든 파일에서 반복된다는 걸 몸으로 익힘.
- main 함수와 if __name__ == '__main__': 블록이 터미널에서 실행될 때만 노드를 띄우는 안전장치라는 걸 다시 확인함.

### [키보드 입력과 차동 구동의 연결]
- pynput 콜백이 단순히 문자를 받는 수준이 아니라, 내부에서 config 속도 상수와 motor_hat API를 묶어서 “사람이 누른 키 → 모터 PWM”까지 직결된 파이프라인이라는 걸 확실히 이해함.
- straight, turn_left, turn_right가 각각 set_differential 비율만 살짝 바꾸는 형태라, 회전 감도는 결국 속도와 비율 조합으로 조정하면 된다는 감각을 얻음.

### [LiDAR → 맵핑 파이프라인 감]
- CustomMsg/포인트 클라우드에서 필요한 점만 추려서 거리·각도로 변환하고, 여러 프레임을 누적해서 노이즈를 줄이는 기본 SLAM 아이디어를 코드 레벨로 다시 정리함.
- 직접 그리는 커스텀 맵핑 코드를 나중에는 slam_toolbox 같은 표준 패키지로 교체해서 더 안정적인 맵을 얻을 수 있다는 그림도 함께 잡음.

### [하드웨어 한계와 파라미터의 의미]
- I2C 주소, 모터 안전 속도, PWM 주파수 같은 값은 “코드 튜닝”이 아니라 “하드웨어 보호 장치”라서 잘못 바꾸면 실제 부품이 망가질 수 있다는 인식을 다시 확실히 함.
- 시연 공간 크기(2.7m x 5.7m)에 따라 속도·장애물 거리·회전 민감도를 조절해야 하고, 이 네 값만 잘 튜닝해도 체감 품질이 크게 달라진다는 걸 이해함.
